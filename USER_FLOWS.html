Path 1 · The gift flow — sender to recipient

1A · Sender creates the gift

Visitor wants to gift a book to a specific person.
Hero — clicks "Gift the Philosophy" CTA→ goSend() shows #sender-section
↓
Sender form — fills name, email, recipient name, recipient email (optional), personal note (optional)all fields hj-suppressed
↓
Submit → POST /api/gift with giftType=giftD1 INSERT into gifts · returns 10-char crypto-random id
↓
If recipient email given → email fires via Resendsubject: "X wants to gift you your life philosophy"
↓
Confirmation screen — shows quiz link, copy button, WhatsApp/email share buttons, "check results later" link with #results/:gid

1B · Recipient takes the quiz

Friend opens the link from email or WhatsApp.
Lands on /#gift/:senderName/:gid→ checkHash() validates gid format (alphanumeric, 16 chars max)
↓
Validate gift → GET /api/gift/:id404? show "this gift link doesn't exist" with self-mode fallback CTA
↓
Quiz landing — shows sender name + personal note (textContent, XSS-safe) + "Discover My Philosophy" CTA
↓
25-question quiz — scores accumulate in sc object, top 3 philosophies computed
↓
Result screen — primary philosophy + quote + top 3 % match. Address form shown (NOT books — surprise preserved)
↓
Submit address → POST /api/resultD1 INSERT into results, UPDATE gifts SET status=completed
↓
Two emails fire
To sender → philosophy + book recommendation + Amazon link + recipient address
Recipient sees → confirmation screen + referral form (kiss-forward)

1C · Sender views results

Sender bookmarks the check link or follows the email.
Opens /#results/:gid→ loadSenderResults(gid)
↓
Fetch → GET /api/gift/:id (returns gift + result if complete)
↓
Two states
If recipient hasn't finished → "Waiting for X to finish..." screen
If complete → philosophy + book card + Amazon button + delivery address (hj-suppressed)
Path 2 · Self-discovery — find your own first

2 · Take the quiz for yourself

Visitor wants their own philosophy match before deciding what to do with it.
Hero — clicks "Find your own first →" (neumorphic ghost CTA next to primary gift CTA)→ goSelf()
↓
Silent gift creation → POST /api/gift with giftType=selfcreates a tracked record so analytics work · sender_email empty so no email fires
↓
Quiz directly — same 25 questions, no sender form interstitial
↓
Result screen — philosophy + quote + top 3, NO address form. Books shown with neumorphic Amazon affiliate buttons. Background: POST /api/result fires silently to save analytics
↓
Pass-it-on block
Primary CTA → "Gift it forward" — routes to sender form (Path 1A) for one-shot gifting
Secondary link → "Schedule it for their birthday" — scrolls to Path 3
Path 3 · Birthday scheduling — set it and forget it

3A · Schedule birthdays

User wants to never forget another friend's birthday.
Lands on #birthday-sectionlinked from hero, footer, self-mode result
↓
Fills "About you" — name + email (where reminders land)
↓
Adds friends — name, email, birthday per row. + Add another friend to add up to 25.date input is hj-suppressed
↓
Submit → POST /api/schedulededuplicates against existing rows for same sender/recipient/birthday · returns 25 max · rate-limit 5/min/IP
↓
Two outcomes
Sender gets → confirmation email listing all scheduled birthdays
UI shows → confirmation screen with each friend in a card, sorted by which fires first, "in N days" countdown
↓
From confirmation screen → "Schedule more birthdays" (resets form, scrolls back) or "Back to home"

3B · Cron fires 14 days before

Daily cron at 03:30 UTC (9:00 AM IST) scans for matching birthdays.
Cron trigger → scheduled() handler in worker fires dailycf cron pattern: 30 3 * * *
↓
Compute target MM-DD = today + 14 days. Special case: Feb 28 in non-leap years also matches Feb 29 birthdays
↓
Query D1 → all scheduled_gifts WHERE birthday_md = target AND last_fired_year < thisYear
↓
For each match → INSERT into gifts (sender_email preserved, gift_type=birthday) → UPDATE scheduled_gifts SET last_fired_year
↓
Two emails per match
To friend → "X wants to gift you a book for your birthday" + quiz link
To sender → "Birthday quiz sent to Y" + tracking link to /#results/:gid
↓
Friend takes the quiz → flows back into Path 1B (recipient flow) → sender gets philosophy + address email
Path 4 · Kiss-forward — viral loop

4 · Recipient becomes sender

After taking a gifted quiz, the recipient pays it forward to one friend.
Recipient confirmation screen shows after submitAddr() completes Path 1B
↓
Referral form — recipient enters their own email + friend's name + WhatsApp numberemail capture added in security pass — without it the loop breaks silently
↓
Submit → POST /api/gift creates new gift (giftType=referral, recipient becomes sender) → POST /api/referral tracks the referral link
↓
WhatsApp opens with prefilled message: "Hey [friend], take this 3-min quiz..." + the new gift link in correct /#gift/Sender/id format
↓
Friend opens link → flows into Path 1B as recipient. Original recipient is now the gifter, gets philosophy + address email when friend finishes.
Endpoints & data layer

API surface · D1 schema · Email triggers

All endpoints rate-limited per IP, CORS locked to thebookofyour.life origin, request body capped at 16KB, all user input sanitized + HTML-escaped before going into emails.

HTTP Endpoints

POST
/api/gift create gift · 20/min/IP · giftType: gift|self|birthday|referral
GET
/api/gift/:id fetch gift + result · 60/min/IP · returns personalNote
POST
/api/result submit quiz outcome · 10/min/IP · 409 if already complete
POST
/api/schedule schedule birthdays · 5/min/IP · dedupes within request & against DB
POST
/api/referral track referral · 10/min/IP · max 5 referrals per call
GET
/api/cron/process-birthdays manual cron trigger · requires ADMIN_TOKEN secret

D1 Tables

gifts
id, sender_name, sender_email, recipient_name, personal_note, status, gift_type, created_at, completed_at
results
gift_id (FK), philosophy_key, philosophy_name, runner_up_1/2, book details, recipient_name/phone/address
referrals
source_gift_id, ref_name, ref_contact, status
scheduled_gifts
sender_name/email, recipient_name/email, birthday, birthday_md (MM-DD index), status, last_fired_year, gift_id

→

→

→
birthday quiz to friend (cron, 14 days before)
→
birthday quiz fired notification to sender (cron, 14 days before)
